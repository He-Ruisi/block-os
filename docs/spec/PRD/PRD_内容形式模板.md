
# 三者统一

```
内容（Content）  →  文字本身，意义，语义
样式（Style）    →  怎么看，视觉表现，标记
模板（Template） →  什么结构，什么组合关系
```

拿你现在的应用举例：

**内容**：「记忆以蛋白质结构的形式存储在大脑中」——这句话的含义，跟它是不是AI生成的、放在什么位置、用什么字体，完全无关。

**样式**：这句话在编辑器里是金色左边框+AI标签，导出时是纯文本，打印时可能是宋体五号字——同一段内容，多种视觉表现。

**模板**：「标题+正文+AI块+正文」这个排列方式——哪些块在一起、先后顺序、是否分章。这是结构。

## 应用内涵

### 第一层：编辑器内部

你现在的块系统其实已经隐约在做这件事了，但还不够彻底：

```
现在的状态：
  内容和样式是混在一起的——AI块的"AI"身份既是内容属性（这段来自AI），
  又是样式属性（金色左边框），还是模板属性（它可以被转为正文改变结构）

应该的状态：
  内容属性：{ text, source: "ai"|"user", timestamp, conversationId }
  样式属性：{ accentColor, showLabel, borderStyle }
  模板属性：{ blockType: "paragraph"|"heading"|"quote", order, group }
```

这三者解耦之后，你能做到的事情完全不同：

### 第二层：预览和导出不只是"去掉标记"

现在你的预览模式是**把样式层剥掉**。但真正的价值在于——**换一套样式和模板**：

```
编辑时：  深色主题 + AI金色标记 + 块级工具栏     → 适合创作
预览时：  浅色主题 + 无标记 + 紧凑排版           → 适合审阅
导出小说：宋体 + 章节模板 + 首行缩进             → 投稿格式
导出博客：Markdown + 引用块样式                  → 发布格式
导出大纲：纯结构 + 层级缩进                      → 思路整理
```

**同一份内容，不同的样式+模板组合，产出完全不同形态的文档。** 这才是"内容与样式分离"在你这个产品里真正的威力。

### 第三层：对"AI块"身份的重新理解

现在AI块的"AI来源"是一个跟样式绑定的属性（有金色边框才看得出来是AI的）。解耦之后：

```
内容层记录：{ source: "ai", conversationId: 42, originalPrompt: "..." }
样式层决定：要不要显示来源标记、用什么颜色、显示多少信息
模板层决定：AI块在导出时是否保留为引用块、还是融入正文
```

用户在编辑过程中改写了AI的内容——这时候它的"来源"信息仍然保留（可以追溯），但样式上可以不再显示AI标记（因为已经面目全非了），模板上可以降级为普通段落。

**身份、表现、结构三者独立变化，互不干扰。**

### 第四层：模板是可以复用和交换的

这是最容易被忽略的。如果你把模板抽象出来：

```
小说模板：  [章节标题] → [正文段落] → [正文段落] → [分隔线] → [下一章]
博客模板：  [H1标题] → [摘要] → [正文] → [引用块] → [正文]
剧本模板：  [场景描述] → [角色名] → [对白] → [动作描写]
```

用户写完内容之后，可以一键切换模板，把同样的素材重新组织成不同文体。甚至可以**把别人的模板导入进来**——社区共享模板。

## 一句话总结

这个口号在你的应用里代表的是：**让用户专注于"写什么"，让系统负责"怎么呈现"和"怎么组织"。** 编辑器不应该是一个固定形态的画布，而应该是一个内容引擎——同一套内容可以被渲染成创作界面、审阅界面、小说稿件、博客文章。这才是从分离中获得的真正自由。


## 交互方案探讨

### 核心理念：两种心智模式

用户在使用这个工具时，本质上在两种模式之间切换——**对话模式**和**写作模式**。对话模式是发散的、探索的，写作模式是收敛的、专注的。界面应该顺应这两种心智状态，而不是把两个工具硬塞在一个固定布局里。

---

### 三种主状态

**1. 对话沉浸态**

整个屏幕就是AI对话。用户在和AI聊天、头脑风暴、存灵感，完全没有写作的压力。灵感素材面板可以从右侧随时展开。这个状态下，编辑器完全不存在——用户甚至不需要知道它在那里。

**2. 写作沉浸态**

整个屏幕就是编辑器。用户进入心流状态，专注写作。AI不是消失了，而是变成一个可随时召唤的"隐形助手"。

**3. 分屏协作态**

就是现在的左右分屏，适合一边聊一边写、频繁交互的场景。

---

### 状态之间的切换

**从对话沉浸到写作**

用户在对话中积累了足够的灵感，准备开始写。这时不需要"打开编辑器"这个操作——而是对话本身自然地"展开"成写作空间。

具体来说：用户点击对话区某个"开始写作"的按钮（或者从灵感面板中选中几条素材后点击"基于这些开始写"），整个对话向左收缩，编辑器从右侧滑入。对话变成左侧窄栏，编辑器占据主区域。如果用户觉得不需要对话了，可以继续收窄对话栏直到完全隐藏，进入写作沉浸态。

**从写作沉浸到对话**

用户在写作中遇到问题，需要AI帮忙。这时不是打开一个完整的对话面板——而是召唤一个轻量的AI小窗。

**从写作态召唤AI小窗**

这个小窗有三种形态，用户可以根据需要切换：

- **悬浮窗**：一个可拖拽、可调整大小的小窗口，浮在编辑器上方。适合偶尔问一两个问题。
- **侧边栏**：贴在编辑器右侧的窄面板，大约占屏幕25%的宽度。适合需要持续对话但又不想离开编辑器的场景。
- **行内气泡**：在编辑器中选中一段文字后，旁边弹出一个小气泡菜单，包含"改写""扩展""润色""解释"等快捷操作。点击后结果直接替换或插入到选区位置。这是最轻量的AI交互，用户甚至不需要打字提问。

这三种形态之间可以渐进升级：行内气泡 → 悬浮窗 → 侧边栏 → 完整对话面板，根据用户需要的交互深度自动或手动切换。

---

### 关键交互细节

**对话到编辑器的"带入"**

当用户从对话沉浸态切换到写作态时，应该有一个"带入"过程——不只是打开一个空白编辑器，而是把对话中存的灵感素材、选中的AI回复，预填到编辑器中。用户进入写作态时，面前不是一个空白页，而是已经有一些素材和骨架的半成品。这个从"对话"到"写作"的过渡，感觉像是把散落的拼图放到了桌面上准备拼装。

**AI小窗的上下文感知**

当AI小窗处于悬浮或侧边栏状态时，它应该自动感知用户当前在编辑器中的位置和选区。用户选中一段文字，小窗自动显示"针对这段文字的快捷操作"；用户光标在某个段落，小窗可以提示"这段和你之前存的某条灵感相关"。AI不是被动等待提问，而是主动提供上下文相关的帮助。

**小窗的"吸附"与"脱离"**

悬浮窗可以自由拖拽，但当拖到屏幕边缘时自动吸附变成侧边栏。反过来，侧边栏可以被"拽出来"变成悬浮窗。这种物理隐喻让用户直觉地管理AI的存在方式。

**键盘驱动的极简切换**

对于重度键盘用户，提供快捷键切换：
- 一个快捷键在对话沉浸和写作沉浸之间切换
- 另一个快捷键在写作态中召唤/隐藏AI小窗
- 选中文字后按快捷键直接发送给AI并附带上下文

这样用户可以完全不碰鼠标，保持心流不被打断。

---

### 状态流转总结

```
对话沉浸 ←→ 分屏协作 ←→ 写作沉浸
  ↑                           ↑
  └──── 灵感面板触发写作 ──────┘
                              ↓
                         AI小窗（悬浮/侧边栏/行内气泡）
```

对话沉浸和写作沉浸之间不应该是二选一的死开关，而是一条连续的光谱。分屏协作是中间态，用户可以通过拖拽分隔线在光谱上自由滑动——对话区越大越偏向探索，编辑区越大越偏向产出。而AI小窗则是在写作沉浸态中保留AI能力的方式，让用户不需要"退出写作"就能获得AI帮助。

---

### 一个可能的用户旅程

用户打开软件，直接进入对话沉浸态。和AI聊了二十分钟关于小说的设定，期间存了五条灵感素材。然后他选中三条素材，点击"开始写作"，界面过渡到分屏态，左侧对话窄栏，右侧编辑器已经预填了他选中的三条素材。他开始写作，写到一半卡住了，选中一段文字按快捷键，行内气泡弹的续写结果直接插入。他继续写，觉得需要深入讨论某个情节，按另一个快捷键，AI小窗以侧边栏形式展开。聊了几轮后觉得够了，按快捷键收起小窗，继续专注写作。最后写完初稿，切换到对话沉浸态，把整篇文章发给AI做全面的反馈和修改建议。
