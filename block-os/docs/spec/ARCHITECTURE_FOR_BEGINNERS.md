# 给小白的项目架构指南

> 如果你是编程新手，这份文档会用最简单的语言解释项目架构的核心概念。

## 🤔 什么是项目架构？

想象你在整理房间：

- **不好的整理方式**：所有东西堆在一起，找东西要翻半天
- **好的整理方式**：衣服放衣柜，书放书架，文具放抽屉

项目架构就是"整理代码和文档的方式"。

## 📁 为什么要按功能分类？

### 场景 1：按时间命名（❌ 不好）

```
我的文件/
├── 2026-04-09-买菜.txt
├── 2026-04-09-工作.txt
├── 2026-04-09-学习.txt
├── 2026-04-10-买菜.txt
└── 2026-04-10-工作.txt
```

**问题**：想看所有"买菜"的记录？要翻遍所有日期的文件。

### 场景 2：按功能分类（✅ 好）

```
我的文件/
├── 买菜/
│   ├── 2026-04-09.txt
│   └── 2026-04-10.txt
├── 工作/
│   ├── 2026-04-09.txt
│   └── 2026-04-10.txt
└── 学习/
    └── 2026-04-09.txt
```

**优势**：想看"买菜"记录？直接打开"买菜"文件夹。

### 在我们的项目中

```
docs/spec/features/
├── editor/      # 编辑器功能都在这里
├── ai/          # AI 功能都在这里
├── block-system/  # Block 功能都在这里
└── storage/     # 存储功能都在这里
```

想了解 AI 功能？直接看 `ai/` 文件夹！

## 📝 为什么每天一个日志文件？

### 场景：记录每天的工作

#### 方式 1：每次创建新文件（❌ 不好）

```
日志/
├── 2026-04-09-09-00-起床.txt
├── 2026-04-09-10-00-吃早餐.txt
├── 2026-04-09-11-00-工作.txt
├── 2026-04-09-14-00-午休.txt
└── ... (一天 10 个文件)
```

**问题**：
- 文件太多，看不过来
- 想看一天的完整流程？要打开 10 个文件

#### 方式 2：每天一个文件（✅ 好）

```
日志/2026-04/2026-04-09.txt

09:00 起床
10:00 吃早餐
11:00 工作
14:00 午休
...
```

**优势**：
- 一个文件看完一天
- 文件数量少，好管理
- 看到完整的工作流程

### 在我们的项目中

```
docs/logs/
└── 2026-04/
    ├── 2026-04-09.md  # 一天的所有工作
    ├── 2026-04-10.md
    └── index.md       # 月度总结
```

## 🤖 为什么 Hooks 要在对话结束时触发？

### 类比：自动保存功能

#### 方式 1：每打一个字就保存（❌ 不好）

```
你在写文章：
"今" → 保存
"今天" → 保存
"今天天" → 保存
"今天天气" → 保存
"今天天气很" → 保存
"今天天气很好" → 保存
```

**问题**：
- 保存太频繁，电脑卡顿
- 产生大量无用的保存记录

#### 方式 2：写完一段后保存（✅ 好）

```
你在写文章：
"今天天气很好，我去公园散步了。" → 写完这段，保存一次
```

**优势**：
- 不频繁打断
- 保存的是完整内容

### 在我们的项目中

#### 改进前（❌ 不好）
```json
{
  "when": { "type": "promptSubmit" }  // 每发一条消息就触发
}
```

你发了 10 条消息 = 触发 10 次 = 产生 10 个重复文档

#### 改进后（✅ 好）
```json
{
  "when": { "type": "agentStop" }  // 对话结束才触发
}
```

一轮对话结束 = 触发 1 次 = 产生 1 个完整文档

## 🏗️ 好的项目结构长什么样？

### 原则 1：清晰的层级

```
block-os/
├── src/           # 代码
├── docs/          # 文档
│   ├── spec/      # 需求（要做什么）
│   ├── guide/     # 指南（怎么做）
│   └── logs/      # 日志（做了什么）
└── .kiro/         # 配置
```

**就像图书馆**：
- 小说区、科技区、历史区（按类型分）
- 不是按书的颜色或大小分

### 原则 2：见名知意

```
✅ 好的命名：
- editor/basic-markdown.md  # 一看就知道是编辑器的基础 Markdown 功能
- logs/2026-04/2026-04-09.md  # 一看就知道是 4 月 9 日的日志

❌ 不好的命名：
- file1.md  # 什么文件？
- temp.md  # 临时文件？
- new-feature.md  # 什么功能？
```

### 原则 3：每个目录有 README

```
features/
├── README.md      # 说明这个目录是干什么的
├── editor/
│   ├── README.md  # 说明编辑器功能
│   └── ...
└── ai/
    ├── README.md  # 说明 AI 功能
    └── ...
```

**就像商场的导览图**：告诉你每个区域是什么。

## 🎯 实际例子：找文档

### 场景：你想了解 AI 对话功能

#### 改进前（❌ 难找）

```
docs/spec/
├── 2026-04-09-phase2.md  # 这是什么？
├── 2026-04-09-ai.md      # 可能是这个？
├── 2026-04-10-chat.md    # 还是这个？
└── 2026-04-11-mimo.md    # 或者这个？
```

你要打开 4 个文件才能找到。

#### 改进后（✅ 好找）

```
docs/spec/features/
└── ai/
    ├── README.md              # 先看这个，了解概况
    ├── chat-integration.md    # AI 对话集成
    └── context-passing.md     # 上下文传递
```

直接打开 `features/ai/` 文件夹，所有 AI 相关的都在这里！

## 🎓 核心概念总结

### 1. 按功能分类，不按时间

**为什么？** 因为你通常是"想找某个功能"，而不是"想找某天的所有东西"。

**类比**：
- ✅ 超市按商品类型分区（食品区、日用品区）
- ❌ 超市按进货日期分区（周一进的货、周二进的货）

### 2. 每天一个日志，追加模式

**为什么？** 因为你想看"一天的完整工作流程"，而不是"碎片化的记录"。

**类比**：
- ✅ 日记本：每天一页，记录一天的事
- ❌ 便签纸：每件事一张，最后一堆纸

### 3. 自动化在合适时机触发

**为什么？** 因为太频繁会产生垃圾，太少会遗漏信息。

**类比**：
- ✅ 每周打扫一次房间（合适）
- ❌ 每分钟打扫一次（太频繁）
- ❌ 一年打扫一次（太少）

### 4. 清晰的目录结构

**为什么？** 因为好的结构让人一眼就懂，不需要解释。

**类比**：
- ✅ 图书馆的分类系统
- ❌ 把所有书堆在一起

## 💡 给小白的建议

### 1. 学习成熟项目

去看看这些项目的文档结构：
- [Vue.js](https://vuejs.org/)
- [React](https://react.dev/)
- [Rust](https://doc.rust-lang.org/)

你会发现它们都是：
- ✅ 按功能分类
- ✅ 清晰的层级
- ✅ 每个目录有说明

### 2. 遵循命名规范

```
✅ 好的命名：
- basic-markdown.md  # 小写，用连字符
- chat-integration.md
- data-model.md

❌ 不好的命名：
- BasicMarkdown.md  # 大写
- chat_integration.md  # 下划线
- dataModel.md  # 驼峰命名
```

### 3. 写 README

每个重要目录都应该有 README.md，解释：
- 这个目录是干什么的
- 里面有什么文件
- 如何使用

### 4. 定期整理

就像整理房间一样，定期整理项目：
- 删除不用的文件
- 合并重复的内容
- 更新过时的文档
