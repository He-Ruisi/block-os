# Bug 修复指南

## 使用 Fix Skill

BlockOS 配置了专门的 `fix` skill 来系统化地处理 bug。

### 激活 Fix Skill

在对话中使用以下方式激活：

```
#fix [描述你遇到的问题]
```

或者直接说：

```
我遇到了一个 bug：[详细描述]
```

### Fix Skill 会做什么

1. **扫描历史问题** 🔍
   - 自动检查 `docs/bugs.md`
   - 查找是否有相似的问题
   - 如果有，参考之前的解决方案

2. **诊断问题** 🔬
   - 收集错误信息
   - 定位错误源头
   - 分析根本原因

3. **实施修复** 🔧
   - 修改代码
   - 运行类型检查
   - 验证修复效果

4. **记录到 bugs.md** 📝
   - 按照统一格式记录
   - 控制在 400 字以内
   - 便于后续查找

## Bug 报告最佳实践

### 好的 Bug 报告 ✅

```
我遇到了一个 bug：

错误信息：
Cannot read property 'schema' of null

触发条件：
1. 打开应用
2. 编辑器区域空白
3. 控制台显示上述错误

环境：
- Chrome 120
- macOS 14.0
- 开发模式

期望行为：
编辑器应该正常显示并可以输入文字
```

### 不好的 Bug 报告 ❌

```
编辑器坏了，修一下
```

## bugs.md 文件说明

### 文件位置
`block-os/docs/bugs.md`

### 文件结构

```markdown
# BlockOS Bug 记录

## 统计
- 总计：X 个
- 本月：Y 个

---

## #001 - 标题 (日期) 🔴
[问题描述、根本原因、解决方案]

---

## #002 - 标题 (日期) 🟡
[问题描述、根本原因、解决方案]
```

### 严重程度标记

- 🔴 **严重**：导致功能完全不可用，需要立即修复
- 🟡 **中等**：影响部分功能，但有 workaround
- 🟢 **轻微**：小问题或优化，不影响核心功能

### 分类标签

Bug 按功能模块分类：
- **编辑器相关**：TipTap、Markdown、格式化
- **AI 对话相关**：API 调用、流式响应、上下文
- **Block 系统相关**：数据模型、链接、引用
- **性能问题**：加载慢、卡顿、内存泄漏
- **UI/样式问题**：布局、颜色、响应式

## 查找相似问题

### 方法 1：使用 grep

```bash
# 搜索关键词
grep -i "editor" block-os/docs/bugs.md

# 搜索错误信息
grep -i "cannot read property" block-os/docs/bugs.md

# 搜索文件名
grep -i "Editor.tsx" block-os/docs/bugs.md
```

### 方法 2：直接阅读

打开 `docs/bugs.md`，按分类查找相关问题。

### 方法 3：使用 Fix Skill

Fix skill 会自动扫描 bugs.md 并告诉你是否有相似问题。

## 记录格式详解

### 问题描述（必填）

```markdown
**问题描述**:
- 现象：用户看到的错误或异常行为
- 触发：什么操作导致的
- 影响：哪些功能受影响
```

**示例**：
```markdown
**问题描述**:
- 现象：编辑器区域空白，控制台报错 "Cannot read property 'schema' of null"
- 触发：页面首次加载时
- 影响：无法使用编辑器功能
```

### 根本原因（必填）

用 1-2 句话说清楚技术层面的根本原因。

**示例**：
```markdown
**根本原因**:
useEditor hook 在组件卸载后仍尝试访问 editor 实例，导致空指针异常。
```

### 解决方案（必填）

```markdown
**解决方案**:
- 修改文件：[文件路径]
- 核心改动：[关键代码变更]
- 预防措施：[如何避免再次发生]
```

**示例**：
```markdown
**解决方案**:
- 修改文件：src/components/Editor.tsx
- 核心改动：添加 editor 存在性检查 `if (!editor) return null`
- 预防措施：所有访问 editor 的地方都先检查是否为 null
```

### 相关问题（可选）

如果有关联的 bug，标注编号。

**示例**：
```markdown
**相关问题**: #003（相同的空指针问题）
```

## 字数控制技巧

每个 bug 记录必须 ≤ 400 字。

### 技巧 1：使用简洁的技术语言

❌ 不好：
```
当用户打开应用程序的时候，编辑器组件没有正确地初始化，
这是因为 React 的 useEditor hook 在某些情况下会返回 null...
```

✅ 好：
```
useEditor hook 返回 null 导致编辑器初始化失败。
```

### 技巧 2：避免冗余描述

❌ 不好：
```
修改文件：src/components/Editor.tsx
在第 15 行添加了一个 if 语句来检查 editor 是否存在，
如果不存在就返回 null，这样可以避免后续代码访问空对象...
```

✅ 好：
```
修改文件：src/components/Editor.tsx
核心改动：添加 `if (!editor) return null`
```

### 技巧 3：重点突出关键信息

只记录：
- 错误现象
- 根本原因
- 核心改动
- 预防措施

不记录：
- 调试过程
- 尝试过的方案
- 详细的代码实现

## 示例：完整的 Bug 记录

```markdown
## #001 - TipTap 编辑器初始化失败 (2026-04-09) 🔴

**问题描述**:
- 现象：编辑器区域空白，控制台报错 "Cannot read property 'schema' of null"
- 触发：页面首次加载时
- 影响：无法使用编辑器功能

**根本原因**:
useEditor hook 在组件卸载后仍尝试访问 editor 实例，导致空指针异常。

**解决方案**:
- 修改文件：src/components/Editor.tsx
- 核心改动：添加 editor 存在性检查 `if (!editor) return null`
- 预防措施：所有访问 editor 的地方都先检查是否为 null

**相关问题**: 无

---
```

**字数统计**：约 180 字 ✅

## 常见问题

### Q: 如果 bug 很复杂，400 字写不下怎么办？

A: 只记录核心信息。如果需要详细说明，可以：
1. 在 bugs.md 中记录摘要（≤ 400 字）
2. 在 `docs/spec/features/` 创建详细的技术文档
3. 在 bugs.md 中添加链接

### Q: 如果是第三方库的 bug 怎么办？

A: 记录 workaround：
```markdown
**根本原因**:
TipTap v2.27.2 的已知 bug（issue #1234）

**解决方案**:
- 临时方案：降级到 v2.26.0
- 长期方案：等待官方修复或切换到其他编辑器
```

### Q: 如果无法复现怎么办？

A: 记录为"待复现"：
```markdown
**问题描述**:
- 现象：[用户报告的现象]
- 触发：未知（无法复现）
- 影响：待确认

**根本原因**:
待复现后分析

**解决方案**:
- 状态：待复现
- 需要信息：[列出需要的信息]
```

## 最佳实践

1. **修复前先查找**：避免重复劳动
2. **记录要及时**：修复后立即记录，避免遗忘
3. **格式要统一**：便于查找和维护
4. **信息要准确**：确保技术描述正确
5. **字数要控制**：保持简洁，≤ 400 字
